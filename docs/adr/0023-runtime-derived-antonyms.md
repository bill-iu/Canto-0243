# Runtime 衍生反義（詞林衍生反義與反義端點鏡射）

領域詞彙：見 [CONTEXT.md](../../CONTEXT.md) § 詞林衍生反義、反義端點鏡射、關係圖、近反義池、效能優化門檻。

## 我們決定

1. **查詢時展開** — **詞林衍生反義**與 **反義端點鏡射** 改為 **查詢時**對當前 head 單點展開；**唔**寫入 `word_relations`，**唔**納入日常 **詞條庫建置命令** inject。
2. **雙消費端** — **近反義模式**（`build_pool`）與 **`!` 展開**（`PoolSnapshot.chars(expand=True)`）皆須覆蓋兩層；展開順序：先 **詞林衍生反義**，再 **反義端點鏡射**。
3. **鄰居權威** — 詞林衍生用 **靜態詞林埠** `get_cilin_synonyms`（cilin-only）；鏡射用 **關係圖** 全源直接近義（DB syn ∪ 靜態合併近義）。
4. **收錄閘** — 衍生候選須在 **收錄字面**（membership）內先納入池。
5. **舊列忽略** — 讀取 `word_relations` 時 **完全忽略** `ant_cilin_exanded`／`ant_syn_mirror` 等衍生 `source` 列；覆蓋只信 runtime。
6. **關係圖快取** — 鄰接採 **進程級 lazy cache**（首次近反義或 `!` 建構，同進程重用）。
7. **快照降級** — TSV 烘焙僅供 maintainer 回歸對照，**唔**再作創作者查詢 SSOT。
8. **效能雙門檻** — 架構優化須同時守 **創作者搜尋 bench** 與 **詞條庫建置時限**（≤10 分）；單邊退化預設不合併，除非另寫 ADR。

**Considered Options**

- **繼續 DB 預寫 + 快照 inject** — build-db 慢（全庫展開）、與 ≤10 分 SLO 衝突。
- **只改 `!`、近反義模式仍靠 DB** — 兩端覆蓋不一致。
- **用鏡射代替詞林衍生** — 全源近義超集，排序語意與 cilin-only 層唔同。
- **每次查詢重建關係圖** — 簡單但重複查詢 P95 退化。
- **就緒閘預熱全圖** — 拖長預載，與就緒閘契約衝突。

**Consequences**

- **詞條庫建置** 更快（唔再 bake／inject ~15 萬衍生列）；現行 ~4 分基線可維持。
- **首次**近反義／`!` 查詢可能付建圖成本；進程 cache 攤平後續查詢。
- 進程記憶體略增（鄰接快取）；與舊 `!` 首次展開同階。
- `expand-antonyms-cilin`／`expand-antonyms-mirror` CLI 保留作 maintainer debug 與回歸匯出。
- 舊 **詞條庫** copy 殘留衍生列 **唔**影響查詢；建議跑一次 **詞條庫建置命令** 清表。
