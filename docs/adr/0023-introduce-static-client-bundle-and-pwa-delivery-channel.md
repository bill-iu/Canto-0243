# ADR-0023: Introduce Static Client Bundle and PWA Delivery Channel

## Status
✅ Accepted

## Context

The Canto-0243 project currently operates primarily as a Python-based CLI application with a server backend for query processing. Users interact with the system through terminal commands, and there is a need to expand accessibility to browser-based environments, particularly for mobile and offline use cases.

To achieve this, we propose introducing a **Static Client Bundle** delivered as a **Progressive Web App (PWA)**. This will enable:
- Browser-based access without installation
- Offline functionality via Service Worker caching
- Mobile "Add to Home Screen" experience
- Cross-platform compatibility (iOS, Android, desktop)

## Decision

We will **implement a React + Vite based PWA client** that:
1. Uses **sql.js** with **sql.js-httpvfs** to load `lyrics.db` as a static asset with chunked/streaming support
2. Ports the core query engine from Python to JavaScript
3. Implements a Service Worker for offline caching of all assets including the database
4. Deploys to **GitHub Pages** as a static site
5. Maintains full offline functionality once loaded

### Technology Stack
- **Framework**: React 18 + TypeScript
- **Bundler**: Vite 5
- **PWA**: vite-plugin-pwa
- **Database**: sql.js + sql.js-httpvfs
- **Deployment**: GitHub Pages (static hosting)

### Architecture
```
client/
├── public/
│   └── lyrics.db              # Static database file
├── src/
│   ├── components/
│   │   └── Search.tsx         # Search UI component
│   ├── db/
│   │   ├── init.ts            # DB initialization with sql.js
│   │   └── query.ts           # Query engine ported to JS
│   ├── hooks/
│   │   └── useDB.ts           # React hook for DB access
│   ├── App.tsx
│   └── main.tsx
├── service-worker.js          # Auto-generated by vite-plugin-pwa
├── vite.config.ts
├── package.json
└── index.html
```

## Consequences

### Positive
✅ **Wider Accessibility**: Users can access Canto-0243 from any browser without Python installation
✅ **Mobile-Friendly**: PWA enables "Add to Home Screen" on iOS and Android
✅ **Offline Capability**: Service Worker caches all assets including lyrics.db for full offline use
✅ **Fast Performance**: Vite provides instant hot module replacement during development
✅ **Small Bundle**: React + Vite produces optimized static bundles suitable for GitHub Pages
✅ **Zero Backend**: Pure static deployment, no server required

### Negative
⚠️ **Bundle Size**: Initial load includes lyrics.db (~10-50MB depending on compression)
⚠️ **Browser Limitations**: SQLite in browser has different performance characteristics than native
⚠️ **Development Complexity**: Maintaining two codebases (Python CLI + JS PWA)
⚠️ **iOS PWA Limitations**: Safari has restrictions on Service Worker storage and background sync

## Alternatives Considered

### 1. Pure Vanilla JavaScript
- **Pros**: Zero dependencies, smallest bundle
- **Cons**: Poor developer experience, manual state management, harder to maintain
- **Decision**: Rejected in favor of React for better maintainability

### 2. Svelte or Vue
- **Pros**: Good alternatives to React
- **Cons**: Team familiarity with React, larger ecosystem
- **Decision**: Rejected in favor of React

### 3. Electron or Tauri (Desktop App)
- **Pros**: Native app experience, full filesystem access
- **Cons**: Requires installation, separate from web access
- **Decision**: Out of scope for this ADR, may be considered separately

### 4. Server-Side Backend API
- **Pros**: Thinner client, centralized query logic
- **Cons**: Requires backend hosting, no offline capability
- **Decision**: Rejected - offline capability is a key requirement

## Risks

1. **Database Size**: lyrics.db may be too large for mobile browsers to cache
   - **Mitigation**: Use compression, consider splitting database

2. **Query Performance**: JS implementation may be slower than Python
   - **Mitigation**: Profile and optimize critical paths, use Web Workers

3. **iOS Service Worker Limitations**: Safari has 50MB storage limit for Service Worker caches
   - **Mitigation**: Implement cache cleanup strategies, warn users if cache fails

4. **SQL.js Memory Usage**: Large queries may consume significant memory
   - **Mitigation**: Implement query pagination, lazy loading
